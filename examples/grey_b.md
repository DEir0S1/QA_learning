# Grey-box

## Чек-лист проверок приложения «Найди и иди»

Чек-лист составлен по методике **Gray-box** и основан **исключительно на требованиях ТЗ**, проанализированных в [req.md](../requirement%20analysis/req.md). В дополнение к пользовательскому интерфейсу тестировщику доступна информация из баз данных, в которых хранится информация о магазинах.

---

## 1. Проверки данных магазинов в БД

- В БД присутствуют записи о магазинах
- У каждой записи магазина заполнено название
- У каждой записи магазина указан тип магазина
- Тип магазина соответствует допустимым значениям из ТЗ (продуктовые, стройматериалы, аптеки и т.д.)
- У каждой записи магазина заполнены координаты (широта, долгота)
- В БД отсутствуют дубли магазинов с одинаковыми координатами и названием
- В БД присутствуют магазины разных типов

---

## 2. Поиск магазина по названию (UI + БД)

- Ввести в поле название магазина, которое точно существует в БД
- Результаты поиска в приложении соответствуют данным из БД
- Количество найденных магазинов в приложении совпадает с количеством записей в БД
- На карте отображаются флажки всех найденных магазинов из БД
- Координаты флажков соответствуют координатам магазинов в БД
- Ввести название магазина, которого нет в БД
- В приложении не отображаются флажки на карте
- Отсутствуют лишние или «фантомные» результаты

---

## 3. Поиск по типу магазина (UI + БД)

- Выбрать тип магазина, для которого в БД есть записи
- В приложении отображаются только магазины выбранного типа
- Количество отображённых флажков соответствует количеству записей в БД
- Выбрать тип магазина, для которого в БД нет записей
- На карте не отображаются магазины

---

## 4. Комбинированный поиск (название + тип)

- Ввести название магазина и выбрать тип, которые совпадают с данными одной записи в БД
- В приложении отображается только соответствующий магазин
- Ввести название магазина, но выбрать несовпадающий тип
- В приложении не отображаются результаты

---

## 5. Кнопка «Поиск» и обработка данных

- Нажатие кнопки «Поиск» инициирует запрос к БД
- При повторном поиске выполняется новый запрос к БД
- Результаты предыдущего поиска не отображаются после нового запроса

---

## 6. Построение маршрута (UI + БД + геолокация)

- Выбрать флажок магазина, координаты которого известны из БД
- Маршрут строится от текущей геопозиции пользователя до координат магазина из БД
- Конечная точка маршрута соответствует координатам магазина в БД
- Выбрать тип магазина
- Маршрут строится до ближайшего магазина выбранного типа
- Выбранный магазин действительно является ближайшим по координатам из БД

---

## 7. Отмена маршрута и состояние данных

- После отмены маршрута данные магазинов не изменяются в БД
- Повторное построение маршрута использует актуальные данные из БД

---

## 8. Повторный запуск приложения и данные маршрута

- Незавершённый маршрут восстанавливается после перезапуска приложения
- Координаты восстановленного маршрута соответствуют данным из БД
- Если данные магазина были изменены в БД, маршрут строится по обновлённым данным

---

## 9. Негативные сценарии, связанные с БД

- Магазин с некорректными координатами в БД не приводит к падению приложения
- Магазин без типа не отображается в результатах поиска по типу
- Магазин без названия не отображается в результатах поиска по названию
