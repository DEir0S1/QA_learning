# REST API

## Теория REST API

### Методы REST API (HTTP Verbs)

Это архитектурный стиль, который говорит серверу какое действие мы хотим совершить над ресурсом (данными).

#### Есть несколько видов запросов

* **GET** — получить данные (например, список пользователей). Безопасный и идемпотентный (сколько ни спрашивай, данные не меняются).
* **POST** — создать новые данные (например, добавить пользователя). Не идемпотентный (повторный запрос создаст еще одного такого же пользователя).
* **PUT** — полностью обновить существующие данные (например, перезаписать информацию о пользователе). Идемпотентный.
* **PATCH** — частично обновить данные (например, изменить только номер телефона пользователя).
* **DELETE** — удалить данные.

### JSON vs XML vs YAML

Это форматы, в которых данные передаются от сервера к клиенту и обратно.

* **JSON (JavaScript Object Notation):** Легкий, читаемый формат. Стал стандартом для REST API.

```json
  {
    "title": "Война и мир",
    "author": "Лев Толстой",
    "pages": 1225,
    "isAvailable": true
  }
 ```

* **XML (eXtensible Markup Language):** Более тяжелый и строгий формат с тегами. Раньше был очень популярен, сейчас чаще используется в старых системах (SOAP) или там, где важна строгая валидация.

```xml
  <book>
    <title>Война и мир</title>
    <author>Лев Толстой</author>
    <pages>1225</pages>
    <isAvailable>true</isAvailable>
  </book>
```

* **YAML (YAML Ain't Markup Language):** Сверхчитаемый формат, который использует отступы. Идеален для файлов конфигурации (Ansible, Kubernetes, Docker Compose).

```yaml
  ---
  title: Война и мир
  author: Лев Толстой
  pages: 1225
  isAvailable: true
```

### Преимущества JSON перед XML и YAML для API

1. **Меньший объем:** JSON легче XML, так как у него нет закрывающих тегов. Это ускоряет передачу данных.
2. **Совместимость с JavaScript:** JSON — это родной для JavaScript формат, что делает его идеальным для веб-приложений.
3. **Читаемость:** JSON читается легче, чем XML, хотя и уступает в этом YAML. Для API это золотая середина между машинной и человеческой читаемостью.
4. **Скорость парсинга:** JSON парсится (разбирается) быстрее, чем XML.

### JSON Schema

Это как чертеж или шаблон, который описывает, как должен выглядеть JSON-документ. С его помощью можно проверять структуру данных: какие поля обязательны, какого они типа и т.д.

* **Пример:** Мы ожидаем, что у пользователя обязательно будет `email` (строка) и `age` (число от 18 до 100).

### Сериализация и Десериализация

* **Сериализация** — это процесс превращения объекта данных в строку JSON, чтобы отправить его по сети.
* **Десериализация** — это обратный процесс. Когда программа получает JSON-строку, она превращает её обратно в удобный объект "Пользователь", с которым можно работать внутри кода.

## 2. Сравнение публичных REST API

1. **JSONPlaceholder** — фейковый API для тестирования и прототипирования. Предоставляет стандартные ресурсы вроде постов, комментариев, пользователей.
    * Ссылка на документацию: [https://jsonplaceholder.typicode.com/guide/](https://jsonplaceholder.typicode.com/guide/)
2. **Petstore (Swagger)** — часть экосистемы Swagger, учебный API для магазина животных. Очень наглядный пример с разными статусами заказов.
    * Ссылка на документацию: [https://petstore.swagger.io/](https://petstore.swagger.io/)

## 4. Анализ JSON-примера

API **Petstore (Swagger)**.

**Анализируемый эндпоинт:** `GET /pet/{petId}` — получение информации о питомце по его ID.
**Ссылка на API:** [https://petstore.swagger.io/#/pet/getPetById](https://petstore.swagger.io/#/pet/getPetById)

### Пример JSON-ответа

Вот как выглядит ответ сервера, когда мы запрашиваем питомца с ID = 1.

```json
{
    "id": 1,
    "category": {
        "id": 1,
        "name": "name1"
    },
    "name": "Max",
    "photoUrls": [
        "string"
    ],
    "tags": [
        {
            "id": 1,
            "name": "name1"
        }
    ],
    "status": "available"
}
```

### Структура JSON: Ключи и типы данных

JSON состоит из пар `"ключ": значение`. Разберем ключи из примера выше и определим их типы данных, опираясь на спецификацию JSON.

| Ключ (Key) | Тип данных (Data Type) | Описание и пример из ответа |
| :--- | :--- | :--- |
| `id` | **Number** (integer) | Уникальный идентификатор питомца. В JSON нет разделения на integer и float, это просто число. Здесь: `1`. |
| `category` | **Object** | Сложный объект, который содержит вложенные данные. Здесь он включает свои собственные ключи `id` и `name`. |
| `category.id` | **Number** (integer) | ID категории животного. Значение: `1`. |
| `category.name` | **String** | Название категории. Строки всегда в двойных кавычках. Значение: `"name1"`. |
| `name` | **String** | Имя питомца. Значение: `"Max"`. |
| `photoUrls` | **Array** | Массив (список) значений. В данном случае массив строк. Элементы массива могут быть любого типа. |
| `tags` | **Array** | Массив объектов. Каждый элемент этого массива — это отдельный объект `tag`. |
| `tags[].id` | **Number** (integer) | Внутри объекта `tag` лежит ключ `id` с числовым значением (`1`). |
| `tags[].name` | **String** | Внутри объекта `tag` лежит ключ `name` со строковым значением (`"name1"`). |
| `status` | **String** | Строка, обозначающая статус питомца. Ожидаемые значения: `"available"`, `"pending"`, `"sold"`. В данном случае `"available"`. |
