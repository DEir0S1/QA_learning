# Базы данных

## 1. Отличия реляционных и нереляционных БД, плюсы и минусы

### Реляционные БД (SQL)

**Реляционные базы данных** хранят данные в виде таблиц со строками и столбцами.  
Связи между данными задаются через ключи (`user_id`, `game_id`, `order_id` и т.д.).

**Примеры:** PostgreSQL, MySQL, MariaDB, Oracle, MS SQL Server.

### Плюсы реляционных БД

- Четкая структура данных.
- Удобно хранить сущности со связями.
- Хорошо поддерживают целостность данных.
- Есть транзакции (ACID), что важно для платежей и покупок.
- Удобны для сложных SQL-запросов.

### Минусы реляционных БД

- Менее гибкие при изменении структуры данных.
- Сложнее масштабировать горизонтально.
- При очень больших нагрузках на запись могут требовать сложной архитектуры.

---

### Нереляционные БД (NoSQL)

**Нереляционные базы данных** хранят данные в других форматах:

- документы,
- key-value,
- wide-column,
- графы.

**Примеры:** MongoDB, Redis, Cassandra, DynamoDB.

### Плюсы нереляционных БД

- Гибкая схема данных.
- Хорошо масштабируются горизонтально.
- Удобны для больших потоков событий и высоких нагрузок.
- Часто быстрее для кэша, логов, временных данных, рекомендаций, сессий.

### Минусы нереляционных БД

- Сложнее поддерживать строгую целостность.
- Не всегда удобно работать со сложными связями.
- Транзакции могут быть ограничены.
- Иногда больше логики приходится реализовывать в приложении.

## 2. Отличия СУБД одного типа: MySQL и PostgreSQL

Обе системы являются **реляционными СУБД**, но между ними есть различия.

### MySQL

- Популярна в веб-проектах.
- Часто проще в освоении.
- Хорошо подходит для стандартных CRUD-операций.
- Удобна для типовых приложений.

### PostgreSQL

- Более мощная и гибкая.
- Лучше подходит для сложных запросов и аналитики.
- Поддерживает расширенные типы данных.
- Часто используется там, где важна надежность и сложная бизнес-логика.

### Вывод

- **MySQL** чаще выбирают для более простых сценариев.
- **PostgreSQL** чаще подходит для сложных систем с большим количеством связей и нетривиальной логикой.

## 4. Гипотеза: какой тип БД является основным для Steam

### Моя гипотеза

Для Steam наиболее вероятен **гибридный подход**:

- **реляционные БД** — для пользователей, покупок, библиотеки, платежей, прав на владение играми;
- **нереляционные БД** — для кэша, рекомендаций, событий, логов, быстрых обновлений статусов, возможно для части отзывов/активности;
- **in-memory / key-value хранилища** — для сессий, онлайн-статусов, временных данных, кэша магазина и друзей.

Steam — это не просто магазин, а большая экосистема, где есть сразу несколько разных типов данных:

1. **Покупки и транзакции**
   - Пользователь покупает игру.
   - Нужно гарантировать, что списание денег и добавление игры в библиотеку происходят корректно.
   - Здесь очень важна транзакционность и консистентность.

2. **Библиотека и права владения**
   - Нужно точно знать, какие игры принадлежат пользователю.
   - Ошибка в этих данных критична.

3. **Социальные функции**
   - Друзья, статусы online/offline, чат, активность.
   - Эти данные часто меняются и требуют быстрого обновления.

4. **Отзывы, рекомендации, активность**
   - Большой объем данных.
   - Возможна высокая нагрузка на чтение.
   - Часто подходят распределенные или кэширующие решения.

5. **Инвентарь и торговая площадка**
   - Есть цифровые предметы и операции обмена/продажи.
   - Для части операций нужна строгая консистентность, для части — быстрый доступ и кэш.

## 5. Разбиение Steam на ключевые функции

### Функция 1. Покупка игры в магазине

### Какие данные используются

- user_id
- game_id
- цена
- скидка
- регион и валюта
- способ оплаты
- order_id
- статус платежа
- дата покупки
- запись о владении игрой

### Какой тип БД больше подходит

**Реляционная БД**

### Почему

- Это финансовая операция.
- Нужна строгая транзакционность.
- Нельзя допустить ситуацию, где деньги списались, а игра не попала в библиотеку.
- Важны связи между заказом, пользователем, игрой и платежом.
- Нужны уникальные и консистентные записи.

---

### Функция 2. Библиотека пользователя

### Какие данные используются

- user_id
- список купленных игр
- дата добавления в библиотеку
- источник получения (покупка, подарок, ключ)
- DLC
- статус установки
- облачные сохранения (метаданные)
- время в игре

### Какой тип БД больше подходит

**Реляционная БД** (с возможным кэшированием)

- Это четкая структура: пользователь ↔ игра.
- Важно точно знать, чем владеет пользователь.
- Нужна целостность данных.
- Хорошо ложится на таблицы связей (`users`, `games`, `user_library`).

---

### Функция 3. Друзья, чат, онлайн-статус

### Какие данные используются

- friend_id / user_id
- список друзей
- статусы online/offline/in-game
- последние сообщения
- уведомления
- текущая игра пользователя
- временные сессии

### Какой тип БД больше подходит

**Гибридный подход:**

- **реляционная БД** — для списка друзей и постоянных связей;
- **нереляционная / in-memory БД** — для статусов, кэша, сессий, быстрых сообщений.

### Почему

- Связи друзей — структурированные и постоянные.
- Онлайн-статусы меняются очень часто.
- Сессии и presence требуют минимальной задержки.
- Чат и уведомления могут требовать высокую скорость записи/чтения.

## 6. Идеи тест-кейсов для проверки выбранных типов БД

### Тест-кейс 1. Проверка транзакционности покупки игры (реляционная БД)

### Цель

Проверить, что покупка игры выполняется как единая транзакция.

### Предусловия

- У пользователя есть аккаунт.
- Игра доступна для покупки.

### Шаги

1. Начать покупку игры.
2. На этапе подтверждения искусственно создать сбой (например, прервать соединение или отменить платеж в последний момент).
3. Проверить библиотеку пользователя.
4. Проверить историю покупок.
5. Повторить покупку успешно.

### Ожидаемый результат

- При неуспешной покупке игра **не появляется** в библиотеке.
- В истории покупок нет некорректной завершенной транзакции.
- При успешной покупке игра появляется в библиотеке **ровно один раз**.
- Нет дублей заказа.

Проверяются свойства **реляционной БД**:

- транзакционность,
- атомарность,
- целостность связанных данных.

---

### Тест-кейс 2. Проверка целостности библиотеки и прав владения (реляционная БД)

### Цель

Проверить, что данные о владении игрой хранятся корректно.

### Предусловия

- У пользователя есть несколько игр в библиотеке.

### Шаги

1. Открыть библиотеку на двух разных устройствах.
2. Убедиться, что список игр одинаковый.
3. Активировать игру через ключ или получить игру подарком.
4. Обновить библиотеку на обоих устройствах.
5. Проверить, что новая игра появилась корректно.
6. Проверить, что одна и та же игра не добавилась дважды.

### Ожидаемый результат

- Список игр синхронизирован.
- Игра появляется после получения права владения.
- Нет дублей.
- Пользователь не теряет доступ к ранее купленным играм.

Проверяется корректность **структурированных данных и связей**:

- пользователь ↔ игра,
- права владения,
- уникальность записей.

---

### Тест-кейс 3. Проверка онлайн-статуса и быстрых обновлений (NoSQL / in-memory)

### Цель

Проверить, что часто меняющиеся данные обновляются быстро и корректно.

### Предусловия

- Пользователи A и B находятся в друзьях.

### Шаги

1. Пользователь A входит в Steam.
2. Пользователь B наблюдает список друзей.
3. Пользователь A запускает игру.
4. Пользователь B проверяет изменение статуса на `in-game`.
5. Пользователь A выходит из игры и затем выходит из клиента.
6. Пользователь B снова проверяет статус.

### Ожидаемый результат

- Статус A быстро меняется с offline на online.
- При запуске игры корректно отображается `in-game`.
- После выхода статусы обновляются без большой задержки.
- Не наблюдается «зависшего» старого статуса.

Это косвенно подтверждает использование **быстрого кэша / in-memory / key-value хранилища** для данных, которые часто обновляются.
